# ProudNet offline protocol (The Game client ↔ local dummy server)

This document describes the **end-to-end wire protocol** used when the game client connects to the repository’s Python dummy server in **offline mode** (`127.0.0.1`, ports **7000 / 27380 / 20009**). It is written for readers who do not have the client binary open beside them.

**Scope:** TCP bytes, framing, transport opcodes, and the parts of the RMI layer we have reverse-engineered. It does **not** claim to be a complete ProudNet 1.7 specification.

**Related:** [proudnet-ida-symbol-map.md](proudnet-ida-symbol-map.md) — `GAME.exe` RVAs, inferred names, and which hooks apply.

**Evidence mix:**

| Source | Trust level |
|--------|-------------|
| Client hooks + IDA on `GAME.exe` (`sub_4BA070`, framing `sub_D84BB0`) | High for structures we marked “IDA-confirmed” |
| Captured traffic during RE (`ctl/logs/runs/…`, `game_netlogs.txt`) | High for ordering and sizes |
| `GITS-FA-emulation/` captures replayed in `server/server/data/*.json` | **Convenience only** — see [§12](#12-opaque-blobs-and-capture-artifacts-do-not-reimplement-blindly) |
| Synthetic fallbacks in `proud_rmi.py` | Low — used only when replay files are missing |

**Relation to other docs:** Investigation notes live in `journals/long/2026-05-27-offline-proud-handshake.md`. This file is the **normative protocol reference** for the offline path.

## 1. Terminology

| Term | Definition |
|------|------------|
| **Proud TCP frame** | One length-prefixed message on the wire. Always starts with magic bytes `13 57`. |
| **Inner payload** | Bytes inside a Proud TCP frame after length decoding. First byte is the **transport opcode**. |
| **Transport opcode** | Single-byte message type for the ProudNet connection FSM (e.g. `0x2F` hello, `0x04` key blob). Distinct from **RMI id**. |
| **RMI message** | Inner payload with transport opcode `0x02`, followed by a 16-bit **RMI id** (little-endian) and a **body**. |
| **RMI id** | 16-bit little-endian method identifier (e.g. `0x3F3E` = NetUserConnectRES). |
| **ENTRY leg** | First TCP connection to port **7000** — master-server / redirect discovery. |
| **GAME leg** | Second TCP connection to port **27380** — lobby session and replayed server→client bursts. |
| **PROBE** | Optional TCP connection to port **20009** — non-Proud echo check. |
| **Burst** | One server→client byte string sent in response to a client `0x25` session message on the GAME leg; may contain **multiple** concatenated Proud TCP frames. |
| **ctl stage** | Client UI milestone reported by `TheGame.dll` (e.g. `shard_choice`, `main_menu`). Reached only after GAME-leg traffic succeeds — not a wire opcode. |

**Endianness:** All multi-byte integers in payloads described here are **little-endian** unless noted otherwise.

**Strings in wire bodies:** UTF-16LE unless noted as ASCII.

## 2. System architecture (offline)

```text
                    ┌─────────────────────────────────────┐
                    │  GAME.exe + TheGame.dll (client)     │
                    │  • ProudNet FSM + RMI dispatch       │
                    │  • Offline remap → 127.0.0.1         │
                    └──────────┬──────────────────────────┘
                               │
         ┌─────────────────────┼─────────────────────┐
         │                     │                     │
         ▼                     ▼                     ▼
   TCP :7000              TCP :27380           TCP :20009
   ENTRY leg              GAME leg              PROBE
   (transport FSM)        (transport FSM +      (raw echo)
                          lobby replay)
         │                     │
         └──────────┬──────────┘
                    ▼
         ┌──────────────────────────┐
         │  server/server/serve.py   │
         │  entry_transport.py       │
         │  game_transport.py        │
         └──────────────────────────┘
```

**Production hosts** (before remap): `137.184.201.52:7000` (entry), `216.131.86.188` / `137.184.201.52:27380` (game). Offline mode rewrites these to `127.0.0.1` on ports **7000**, **27380**, and **20009** (`include/game/server_override.hpp`).

**Important:** The **redirect message (`0x0A`) carries an IP address only**, not a port. The client learns the game port **27380** from ENTRY **server-info** frames (`0x02`), not from the redirect blob.

**Verified client outcome (run `018_5cb743bb`):** After the sequence below, ctl stages progress to `shard_choice` then `main_menu`. The in-game server list still shows strings from **captured** server-info / lobby data (e.g. names containing **“TOM LOCAL EMU”**, character **“Tom_Neverwinter”**) — evidence that much of the visible UI text is replayed verbatim, not generated by our config alone.

## 3. Layer: Proud TCP frame

Every Proud leg uses the same framing (client `sub_D84BB0` / `sub_D84970`; server `proud_frame.py` / `include/game/net/proud_frame.hpp`).

### 3.1 On-wire layout

| Offset | Size | Field | Description |
|--------|------|--------|-------------|
| 0 | 2 | `magic` | Fixed `0x57 0x13` (bytes **`13 57`**) — uint16 value `0x5713` LE. |
| 2 | 1 | `len_tag` | Width of length field: `1`, `2`, or `4`. |
| 3 | `len_tag` | `payload_len` | Unsigned length of **inner payload** only. |
| 3+len_tag | `payload_len` | `payload` | Inner transport message (see §4). |

### 3.2 Length encoding (`len_tag`)

| `len_tag` | Length bytes | Used when |
|-----------|--------------|-----------|
| `1` | 1 byte | `payload_len ≤ 0x7F` (implementation uses tag 1 when `length + 128 ≤ 0xFF`) |
| `2` | 2 bytes LE | Medium frames |
| `4` | 4 bytes LE | Large frames (e.g. big `0x25` session blobs) |

**Parser rule:** Buffer until `3 + len_width + payload_len` bytes are available, then consume one frame. Multiple frames may arrive in one `recv()`.

### 3.3 Example (client hello)

Canonical ENTRY hello (9 bytes total):

```text
13 57 01 05  2F 0F 00 00 40
│  │  │  │   └─ inner payload (5 bytes)
│  │  │  └──── payload_len = 5
│  │  └─────── len_tag = 1
│  └────────── magic
```

## 4. Layer: Inner payload (transport)

| Offset | Size | Field |
|--------|------|--------|
| 0 | 1 | `opcode` |
| 1 | rest | `data` — opcode-specific |

If `opcode == 0x02`, `data` is an **RMI message** (§5).

### 4.1 Transport opcodes (used in offline path)

| Opcode | Name | Typical direction | ENTRY (:7000) | GAME (:27380) |
|--------|------|-----------------|---------------|---------------|
| `0x2F` | Hello | C2S | ✓ | ✓ |
| `0x04` | Key blob | S2C | ✓ | ✓ |
| `0x05` | Key response | C2S | ✓ | ✓ |
| `0x06` | ACK | S2C | ✓ | ✓ |
| `0x07` | Auth | C2S | ✓ | ✓ |
| `0x0A` | Redirect | S2C | ✓ | ✓ |
| `0x02` | Server info / RMI carrier | S2C | ✓ (server list) | ✓ (inside replay) |
| `0x1B` | Ping | C2S | — | ✓ |
| `0x1D` | Pong | S2C | — | ✓ |
| `0x1C` | Keepalive | both | C2S silent on ENTRY; S2C periodic | S2C reply to C2S; periodic S2C |
| `0x25` | Session | C2S | ✓ | ✓ (triggers lobby bursts) |

Other opcodes exist in ProudNet generally; this table lists what the dummy server handles.

## 5. Layer: RMI message (`opcode == 0x02`)

| Offset | Size | Field |
|--------|------|--------|
| 0 | 1 | `0x02` |
| 1 | 2 | `rmi_id` (uint16 LE) |
| 3 | n | `body` |

Client summarization: `include/game/net/proud_message.hpp` (`ProudMessage::summarize_payload`).

### 5.1 RMI ids on the connect / lobby path

| RMI id (hex) | Decimal | Name (informal) | Body size (offline server) | Understanding |
|--------------|---------|-----------------|----------------------------|---------------|
| `0x3F3E` | 16190 | NetUserConnectRES | **0x180** (384 B) | **IDA-confirmed** layout (§8) |
| `0x3E8E` | 16014 | NetConnectRES | 64 B in stub | **Not understood** — placeholder zeros |
| `0x3E99` | 16025 | NOTIFY | 6 B stub + arg `15060` | **Not understood** — placeholder |
| `0x00D2` | 210 | “d23e tick” (post-init) | From capture | **Opaque** — periodic heartbeat in replay |

Lobby replay files contain **many additional** RMI ids inside concatenated frames; only `0x3F3E` is fully described field-by-field in this repo.

## 6. End-to-end sequence

### 6.1 ENTRY leg (port 7000)

| Step | Client → Server | Server → Client | Notes |
|------|-----------------|-----------------|-------|
| E1 | Proud frame, inner `0x2F` + 4-byte tail | — | Tail bytes observed: `0F 00 00 40` (purpose not fully reverse-engineered). |
| E2 | — | Inner `0x04` + **key blob** (§12.1) | Static blob; enables client `0x05`. |
| E3 | Inner `0x05` + ciphertext | — | **Opaque** client key material; server does not decrypt in Python. |
| E4 | — | Inner `0x06` (empty ACK) | 5-byte framed message. |
| E5 | Inner `0x07` + auth blob | — | **Opaque** credentials/session material. |
| E6 | — | Inner `0x0A` + redirect body (§7) | IP = `REDIRECT_HOST` (default `127.0.0.1`). |
| E7 | Inner `0x1C` keepalive | — | Server sends **no** reply on ENTRY. |
| E8 | Inner `0x25` + large session blob | — | **Opaque** client session state. |
| E9 | — | Two full Proud frames, inner `0x02` **server-info** (§7.2) | Advertises game port / shard list UI strings. |
| E10 | — | S2C `0x1C` every **3 s** | Background keepalive until disconnect. |

After E9 the client opens **PROBE :20009** (optional) and **GAME :27380**.

### 6.2 GAME leg (port 27380)

| Step | Client → Server | Server → Client | Notes |
|------|-----------------|-----------------|-------|
| G1–G6 | Same as E1–E6 | Same as E2–E6 | Independent TCP connection; same transport FSM. |
| G7 | `0x1C` | `0x1C` | Unlike ENTRY, server **echoes** keepalive. |
| G8 | `0x1B` ping (optional) | `0x1D` pong | Pong body = ping prefix + **uint32** timestamp ms (LE). |
| G9 | `0x25` session (1st) | **Burst 0** — verbatim bytes (§12.3) | 7323 B in shipped `lobby_replay.json`. |
| G10 | `0x25` (2nd) | **Burst 1** | 2920 B. |
| G11 | `0x25` (3rd) | **Burst 2** | 1193 B; contains captured UTF-16 **“Tom_Neverwinter”** etc. |
| G12 | `0x25` (later) | — | No reply after bursts exhausted. |
| G13 | — | **Post-init** schedule: 9 frames (§12.4) | Delayed 0.15 s + i×0.08 s after last burst. |
| G14 | — | S2C `0x1C` every **5 s** | From GAME connect. |
| G15 | — | `0x02` / RMI `0xD2` every **10 s** | If found in post-init capture. |

**Fallback:** If `lobby_replay.json` is missing, first `0x25` returns synthetic `0x3F3E` + stub follow-ups (`config.SEND_RMI_ON_GAME_25`) — insufficient for `main_menu` in practice.

### 6.3 PROBE (port 20009)

| Condition | Behavior |
|-----------|----------|
| First byte of TCP chunk is `0x2F` | Server echoes **entire chunk** unchanged. |
| Otherwise | Ignored. |

**No Proud framing** on this port.

### 6.4 ctl stages (observable outcome, not wire)

| Stage | Meaning |
|-------|---------|
| `connecting_to_server` | Client called `TCPSocket::Connect` toward entry port 7000. |
| `shard_choice` | `CGameServer::onPreProcess` begin (shard / server picker UI). |
| `main_menu` | Same hook, end — main menu wired. |

Requires GAME-leg replay success, not merely ENTRY handshake.

## 7. Partially understood server-built messages

### 7.1 Redirect (`opcode 0x0A`, payload after opcode byte)

Built by `build_redirect_frame()` — layout matches FA-EMU **shape**; token bytes are random each connection.

| Offset | Size | Field | Source |
|--------|------|--------|--------|
| 0 | 4 | `conn_id` | uint32 LE; monotonic from 1000 |
| 4 | 16 | `session_token` | Random bytes |
| 20 | 4 | `flags` | Fixed `01 00 01 01` (meaning **not** verified) |
| 24 | 1 | `ip_len` | ASCII length |
| 25 | `ip_len` | `ip` | ASCII host (e.g. `127.0.0.1`) |
| 25+len | 2 | `trailer` | Random bytes (meaning **not** verified) |

**No port field.** Client must already know **27380** from server-info.

### 7.2 ENTRY server-info (two full Proud frames, inner `0x02`)

Each frame is **80 bytes** on the wire, assembled as:

```text
[ captured_prefix_28_bytes ][ server_name_36_bytes_UTF16LE ][ captured_postfix_16_bytes ]
```

| Piece | Source | Notes |
|-------|--------|-------|
| `captured_prefix` | Hex constants `_ENTRY_INFO_1_PREFIX` / `_ENTRY_INFO_2_PREFIX` | Includes magic, length, inner `02`, and fixed fields (session cookie `89b8c934…`, port-like `f4 6a` → **0x6AF4 = 27380**, sub-opcodes `0x6942` / `0x426b`). |
| `server_name` | **Computed** from `config.SERVER_NAME` (default **`TOM LOCAL EMU`**) | Max **17** UTF-16 code units; padded to 36 bytes. |
| `captured_postfix` | `_ENTRY_INFO_1_POST` / `_ENTRY_INFO_2_POST` | Fixed trailing bytes. |

Changing only `SERVER_NAME` updates part of the UI label; **prefix/postfix still embed FA-EMU capture bytes** (including unrelated session ids).

### 7.3 Pong (`opcode 0x1D`)

```text
body = ping_data || uint32_le(floor(time_ms))
```

Wrapped in a standard Proud frame.

## 8. NetUserConnectRES (RMI `0x3F3E`) — IDA-confirmed body

**When used:** Inside GAME lobby replay bursts, and as synthetic fallback. **Not** sent as the first reply to `0x2F` on port 7000 in the working offline path.

**Body size:** **384 bytes (`0x180`)**. Handler: `sub_4BA070` (`onNetUserConnectRES`).

| Offset | Size | Field | Type | Semantics |
|--------|------|--------|------|-----------|
| `0x000` | 2 | pad | — | Ignored |
| `0x002` | 2 | `error_code` | uint16 | **0 = success**; `30213` / `30214` → auth failure path |
| `0x004` | 4 | `farm_select_id` | uint32 | Stored in out-game manager |
| `0x008` | 4 | `session_uid` | uint32 | Passed to `sub_890AA0` — **not** account id |
| `0x00C` | 4 | pad | — | |
| `0x010` | 76 | `name` | wchar[38] | Max **37** visible chars + NUL |
| `0x05C` | 4 | `field_5c` | uint32 | Farm selector — **semantics unclear** |
| `0x060` | 4 | `field_60` | uint32 | Player slot table index — **semantics unclear** |
| `0x064` | 0x88 | `unk_block` | bytes | Zeros work in tests |
| `0x0EC` | 4 | **`account_id`** | uint32 | **Required** — stored netmgr+0x88 / +0x8C |
| `0x0F0` | 120 | `farm_slots[3]` | 3×40 B | **Layout not fully reverse-engineered** |
| `0x16C` | 4 | `num_char_slots` | uint32 | Character slot count |
| `0x170` | 16 | pad | — | |

Reference: `include/game/net/net_user_connect_res.hpp`, `server/server/proud_rmi.py`.

## 9. Client requirements (offline)

For the sequence above to complete on a developer machine:

1. **`TheGame.server_ip`** or **`THEGAME_SERVER_IP`** set (e.g. `127.0.0.1`) so production IPv4 addresses remap on ports **7000**, **27380**, **20009**.
2. **All `connect` IAT slots** in `GAME.exe` and `TheGame.dll` hooked, plus ProudNet **`w_connect_3`**, so connects cannot bypass remap (`src/main.cpp`, `src/hooks/w_connect_3.cpp`).
3. **`pn_select_sync` must not overwrite** the GAME socket handle with the ENTRY socket after `:27380` connect (`socket_trace.cpp`).
4. Dummy server listening on **all three ports** (`just ensure-serve`).

## 12. Opaque blobs and capture artifacts (do not reimplement blindly)

This section lists material that **must stay byte-identical** (or be replaced only with a **fresh capture from the same client build**) because we **do not** know how to construct it from first principles. **Do not treat `GITS-FA-emulation/` documentation or comments as ground truth** — it was a third-party emulation effort; we imported captures because they happened to unlock the UI, not because they were validated against Nettention specs.

### 12.1 `PN_KEY_BLOB` (S2C inner payload of `0x04`)

| Property | Value |
|----------|--------|
| Size | **182 bytes** after opcode `0x04` (**188 bytes** framed with tag `02`) |
| Location | `server/server/proud_rmi.py` — constant `PN_KEY_BLOB` |
| Origin | FA-EMU / live capture hex |
| Why opaque | ASN.1/RSA/DH hybrid structure; server does not generate per session — client must accept this **exact** modulus/exponent packaging or the `0x05` exchange fails. |
| If wrong | Client never sends `0x05`, or disconnects before `0x07`. |

### 12.2 Client C2S blobs: `0x05`, `0x07`, `0x25`

| Message | Observation |
|---------|-------------|
| `0x05` key response | Variable length (~155 B in logs); encrypted relative to key blob — **not decrypted** by Python server. |
| `0x07` auth | Variable (~27 B on ENTRY); semantics unknown. |
| `0x25` session | Large (~1.5 KB+); opaque session ticket / account state. |

Server handlers: **accept any payload** and advance FSM — correctness depends on client, not server parsing.

### 12.3 `lobby_replay.json` (GAME leg bursts)

| Property | Value |
|----------|--------|
| Format | JSON array of **3** hex strings |
| Shipped sizes | **7323**, **2920**, **1193** bytes per burst |
| Structure | Each string is **concatenated complete Proud TCP frames** — not parsed field-by-field in code |
| Origin | Copied from FA-EMU / live-server capture |
| Visible side effects | UI shows **“Tom_Neverwinter”**, **“[SYSTEM]”**, shard/GP strings from emulator era — **not** from `SERVER_NAME` alone |
| Why opaque | Dozens of RMI messages with unknown bodies; inter-frame timing and ordering matter. |
| If wrong | Stuck at `connecting_to_server`, IOCP errors, or crash — even when ENTRY succeeds. |

**To change branding in the main menu:** replacing `SERVER_NAME` is **insufficient**; you must re-capture or hex-edit inside these bursts.

### 12.4 `lobby_postinit.json` (post-burst schedule)

| Property | Value |
|----------|--------|
| Format | JSON array of **9** hex strings (separate Proud frames) |
| Origin | FA-EMU “live lobby” capture |
| Contains | Donation/community UTF-16 strings, miscellaneous `0x02` RMI traffic, optional **`0xD2`** tick frame |
| Why opaque | Same as bursts — replay-only. |
| If wrong | May still reach `main_menu` briefly or miss periodic lobby updates — behavior untested for partial edits. |

### 12.5 ENTRY server-info prefix / postfix (`_ENTRY_INFO_*`)

| Property | Detail |
|----------|--------|
| What we change | Only **36-byte UTF-16 name** (`SERVER_NAME`) |
| What is frozen | **28-byte prefix** and **16-byte postfix** per frame, including session cookie `89b8c934…`, constants `f4 6a`, `29 4e`, sub-opcode tags |
| Why opaque | Full semantic map of server-list entry fields not reverse-engineered — only port-like `0x6AF4` inferred |
| FA-EMU smell | Default name **`TOM LOCAL EMU`** matches emulator branding |

### 12.6 Redirect `flags` and 2-byte trailer

Fixed / random in implementation; **no IDA proof** that values are ignored. Treat as **capture-faithful** when porting.

### 12.7 Stub RMI follow-ups (`0x3E8E`, `0x3E99`)

| RMI | Status |
|-----|--------|
| `0x3E8E` NetConnectRES | 64 zero bytes — **WIP**, not used in successful replay path |
| `0x3E99` NOTIFY + arg `15060` | 6-byte stub — **WIP** |

Real session uses payloads **inside lobby replay**, not these stubs.

### 12.8 Farm slots and `field_5c` / `field_60` in `0x3F3E`

Zeros succeed for reaching `main_menu` in offline test, but farm/shard UI may be wrong for gameplay. **Do not assume** zero is always valid on live servers.

### 12.9 What is *not* opaque (safe to implement from this doc)

- Proud TCP framing (§3)
- Transport opcode FSM order on ENTRY and GAME (§6)
- Redirect **layout** (§7.1) with generated token/id
- Pong timestamp (§7.3)
- NetUserConnectRES **field offsets** and success `error_code == 0` (§8)
- PROBE raw echo (§6.3)

## 13. Replacing FA-EMU artifacts responsibly

1. Capture a new trace from a **known-good client build** talking to any server that reaches the desired UI (wireshark or `ctl` netlogs + server `wire_*.log`).
2. Split into Proud frames; validate magic `13 57` boundaries.
3. Replace **one artifact at a time** (`PN_KEY_BLOB`, then bursts, then post-init).
4. Re-run `just ctl launch-offline` and confirm `game_states` include `main_menu`.
5. Update hex constants in `server/server/data/` or `proud_rmi.py` — keep old captures in git only if license permits.

Until that work is done, treat anything that makes the client display **“Tom’s server”**, **“Tom_Neverwinter”**, or **“TOM LOCAL EMU”** as a **smoke test artifact**, not as proof that the protocol is fully understood.
